You can get Impact maintenance informations starting, for example, from requests like:

  • Database column size changing
  • Database column inserting/deleting
  • Copy book item modifying
  • New map field
  • Map field changing

Inpact will analyze all elementary data item transformations to get all sources involved directly or indirectly by the change request.

Afterwards, retrieved the list of sources and rows inside sources to be changed, an evaluation of the estimated maintenance time can be done by means of customer dependent average remediation time.

Inpact can be useful in several cirìcumnstances to the project leader, the analyst and the programmer.

Your'e allowed to budget the activity through the preliminary evaluation of the maintenance effort, to support the process analysis, or as operative support to the actual source changing activity.

The system integrator can use Impact to get boundary subsystem objects too.
 

Before any maintenance task it's necessary to evaluate the overall cost of the maintenance activity.

Often what that seems a little code changing actually involves extensive manipulation of hundreds of sources, with overshoot of costs and budget.
So it's fundamental to evaluate the maintenance activities and therefore the cost, through impact analysis functions.

Of course it's possible manually to query the Amrita database, showing objects, relationships, origin relationships and where-used, to get all of required informations.

Impact does this job automatically, and allows you to know in advance all objects source rows to be modified.

Your'e noticed that informations obtained by Impact are safe as the assesment process pruduced an exaustive set of application objects and relationships, dynamically generated too.
