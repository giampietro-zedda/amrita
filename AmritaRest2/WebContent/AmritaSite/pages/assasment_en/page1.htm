Pgm Level Process

Programs must be already analyzed by Source Analysis Process and therefore any processing is done by means of the java object descriptor and sources are not necessary anymore.
The scope of this process type is always the single program regardless connections with other programs.
You are allowed to enable specific processing like:

1) Metrics basic & complexity like McCabe and others
2) Dynamic code
3) Dead code
4) Where used copy and fields
5) Where used entity and entity fields
6) Pecompiler Cics/Sql instruction analysis

System Level Process

The scope of this processing is not the single program but the entire system to solve what can't be solved in the single program. The system level process works with available database informations to aggregate and to get further informations.
You are allowed to enable specific processing like:

1) Metrics aggregations and quality indexes computing
2) Dynamic code spreaded

A detailed report of the application quality status, allows you to decide for the fixing of potential errors, reducing the fault risk, or simply getting advantage using available informations, in the legacy applications maintenance process.

Library Scan Process

All sources object must be detected and classified before any further processing.
Assesment automatically detect the source type in the library and puts the objects names and relationships on the Amrita database.

Sources Analysis Process

Sources classified by Library Scan are analyzed by specific analyzers enabling only desired analysis options. Basic metrics measures, like dimensional LOC, CLOC and so on are always detected. A java object is generated for any source

  

Assesment Key points

  • More than 100 quality measures
  • SQALE quality system based on violations and remediation costs
  • Indicators for complexity, maintenability (MI), dynamic code, dead code
  • Classic complexity measure by McCabe and Halstead
  • Structural and technical complexity measures, Fan-In, Fan-Out etc.
  • Functional general complexity measures as number of objects, relationships, internal calls and external calls
  • Remediation costs by violation and established quality constraints
  • About 160 violation types detected in program sources
  • List of objects analyzed and dynamically potentially referenced
  • List of objects analyzed and never referenced, neither dynamically
  • Unused libraries objects, tu put in the garbage list
  • Exaustive set of relations between object, even if dynamically generated
  • Source row origin of each relation
  • Java object for each source analyzed, with methods to programmatically
    process assesment results, if it needs to specific projects
  • Structured informations for where-used, impact-analysis, data-linage,
    automatic maintenance, reverse, re-engineering and documentation
Sources parsing is only the first step of the analysis process and knowledge detecting
